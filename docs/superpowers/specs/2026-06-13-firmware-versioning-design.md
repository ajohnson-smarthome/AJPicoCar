# Версионирование прошивок + авто-детект обновления — Design

**Дата:** 2026-06-13
**Статус:** дизайн утверждён (обсуждение в чате), готов к плану

## Цель

Каждый билд получает **порядковый номер** (авто, монотонный), версия — **семантика + номер билда**
(`v1.2+246`). OTA сравнивает версии **численно по номеру билда**, так что по данным с GitHub апп
однозначно понимает «вышло обновление». Релизы режутся скриптом (запускаемым в т.ч. ассистентом по запросу).

## Текущее состояние (база)

- Версия = `git describe --tags` → сейчас `v1.0-109-g0724c83`. `/status` `fw` несёт её.
- OTA (iOS `UpdateClient`): `normalize(release.tag) != normalize(status.fw)` → «обновление». Сравнение
  по **равенству строк**, без числового порядка.
- Один тег `v1.0`, один GitHub-релиз `v1.0`. `git rev-list --count HEAD` = 246 (монотонный счётчик).

## Схема версии

| Часть | Источник | Пример |
|---|---|---|
| Семантика (мажор.минор) | `version.txt` в корне (версионируется) | `1.2` |
| Номер билда | `git rev-list --count HEAD` (авто) | `246` |
| **Полная версия** (в прошивке/`/status`/теге релиза) | вычисляется | **`v1.2+246`** |

- Формат `v1.2+246`: влезает в лимит `esp_app_desc` version (32 символа); номер билда парсится после `+`.
- Номер билда монотонно растёт с каждым коммитом → новый релиз всегда имеет больший номер (даже без
  бампа семантики). Семантика — только для людей.

## Компоненты

### 1. `version.txt` *(new)* — корень репо
Содержит только семантику, напр. `1.2` (без `v`, без номера билда). Человек/ассистент бампает на вехах.

### 2. Корневой `CMakeLists.txt` — встраивание
До `project()`:
- Прочитать `version.txt` → `SEMVER`.
- `execute_process(git rev-list --count HEAD)` → `BUILD_NUM` (фоллбэк `0`, если git недоступен).
- `set(PROJECT_VER "v${SEMVER}+${BUILD_NUM}")`.
Тогда `esp_app_get_description()->version` = `v1.2+246` автоматически; `/status` `fw` уже его несёт
(телеметрия по WS / бутстрап — без изменений в формате, просто значение становится с номером).

⚠ Нюанс: CMake кэширует configure; номер билда обновляется при реконфигурации (clean/изменение CMake).
Для **релиза** это не проблема — `release.sh` делает чистую сборку. Для локальной разработки — номер
освежается на `idf.py fullclean`/реконфиг (приемлемо: критичен номер именно релизного `.bin`).

### 3. `tools/release.sh` *(new)* — неинтерактивный, запускаемый ассистентом по запросу
1. `SEMVER=$(cat version.txt)`, `BUILD_NUM=$(git rev-list --count HEAD)`, `VER="v${SEMVER}+${BUILD_NUM}"`.
2. Проверки: дерево чистое (иначе номер билда не совпадёт с тем, что в релизе); на ветке `main`.
3. Чистая сборка: `idf.py fullclean && idf.py build` (с python3.13-шимом + export.sh).
4. `gh release create "$VER" build/esp32-p4-car.bin --title "v${SEMVER} (build ${BUILD_NUM})" --notes "<...>"`.
5. `--dry-run` флаг: печатает план (версию, тег, что зальёт) **без** сборки/публикации.
- Сборка — не флеш (безопасно). Публикация на GitHub — наружу, выполняется с авторизации пользователя
  («выпусти релиз»). Ассистент не публикует самовольно.

### 4. OTA — численное сравнение
- **iOS `UpdateClient`**: новая чистая функция `buildNumber(_ version: String) -> Int?` — берёт подстроку
  **после первого `+`** и парсит как Int (`v1.2+246` → `246`); `nil`, если `+` нет или хвост не число.
  Решение об обновлении: оба номера есть и `latestBuild > runningBuild` → «доступно». Заменяет
  `normalize`-равенство.
- Тег релиза = полная версия `v1.2+246`; апп парсит номер из тега релиза и из `status.fw`.
- **`FirmwareView`**: показывает «Текущая v1.2+246», «Доступно v1.3+250» (семантика для человека, номер
  для сравнения). Если у running нет номера (старая прошивка) — фоллбэк на прежнее строковое сравнение,
  чтобы не сломать апгрейд со старья.

## Тестирование

- **iOS host-тест** (swiftc + XCTest-зеркало): `buildNumber("v1.2+246") == 246`, `buildNumber("v1.0") == nil`,
  сравнение `246 < 250`, битые строки.
- **Прошивка:** сборка → `/status` `fw` == `v<version.txt>+<git count>` (проверить значением).
- **`release.sh --dry-run`:** печатает корректные версию/тег/файл без публикации.
- **Симулятор:** мок `/status` отдаёт `fw` с номером + мок «latest release» с бОльшим номером →
  `FirmwareView` показывает «Доступно обновление»; равный номер → «Актуальная версия».

## Вне объёма

- Механика заливки OTA (`/ota`, dual-partition, rollback) и сам процесс прошивки — без изменений.
- CI/GitHub Actions (выбран ручной скрипт A).
- Changelog-автоматизация (ноты релиза пишутся вручную/ассистентом при выпуске).
