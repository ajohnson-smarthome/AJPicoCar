# OTA-обновление прошивки через мобильный апп — Design

**Дата:** 2026-06-09
**Статус:** дизайн утверждён, готов к плану

## Цель

Обновлять прошивку машинки «по воздуху» из iOS-приложения: апп берёт `.bin` из **GitHub Releases**
(когда телефон в интернете) и заливает его на машинку по локальному Wi-Fi (`POST /ota`), которая пишет
образ в свободный OTA-раздел, проверяет, перезагружается и при сбое откатывается. Без проводов после
разовой установки OTA-разметки.

## Поток (пользователь)

1. (Телефон в интернете) Апп дёргает **GitHub Releases API** репо `ajohnson-smarthome/AJPicoCar`
   (публичный, без токена) → последний тег + asset `esp32-p4-car.bin`; сравнивает тег с версией машинки
   из `/status` (`fw`). Новее → «Доступно обновление X → Y», качает `.bin` в кэш.
2. Апп просит подключиться к Wi-Fi **`ESP32-Car`** (интернет для заливки не нужен).
3. Апп заливает `.bin` на машинку (`POST http://192.168.4.1/ota`, стрим) с прогресс-баром →
   машинка пишет в свободный OTA-раздел → валидирует → перезагружается → апп переподключается,
   читает `/status`, показывает новую версию.

## Пререкизиты

- **Репо на GitHub публичный** — ✅ готово (`https://github.com/ajohnson-smarthome/AJPicoCar`).
- **GitHub Releases:** каждый релиз — git-тег `vX.Y` + asset `esp32-p4-car.bin` (артефакт `idf.py build`).
  Первый релиз `v1.0` нужно выложить, чтобы апп было что качать.
- **Разовый wired-flash dual-OTA разметки** (см. ниже).

## Прошивка

### Разметка и откат
- `partitions.csv` (custom): `nvs`, `phy_init`, `otadata`, **`ota_0`**, **`ota_1`** (два app-слота ~1.5 МБ;
  флеш 4 МБ XIAO C6 хватает). `sdkconfig.defaults`: `CONFIG_PARTITION_TABLE_CUSTOM=y`,
  `CONFIG_BOOTLOADER_APP_ROLLBACK_ENABLE=y`.
- **Разовая прошивка по USB** этой разметки (`idf.py flash`), дальше — по воздуху.
- **Откат:** после OTA-загрузки новый образ в состоянии «pending verify»; когда поднялись Wi-Fi+HTTP и всё
  ок — `esp_ota_mark_app_valid_cancel_rollback()`. Если образ упал раньше — бутлоадер откатывается на
  предыдущий слот (защита от кирпича).

### Версия из сборки
- `/status` `fw` = `esp_app_get_description()->version` (вместо хардкода `"1.0"`). Версия берётся из
  git-тега через `PROJECT_VER` (ESP-IDF авто-выводит из `git describe`, либо `version.txt`). Так
  `/status` и тег релиза сопоставимы.

### Эндпоинт `/ota`
- `main/ota_api.{c,h}`: `POST /ota` на общем httpd (`http_server_get_handle()`):
  1. `car_stop()` (моторы стоп на время прошивки);
  2. `esp_ota_begin(esp_ota_get_next_update_partition(NULL), OTA_SIZE_UNKNOWN, &handle)`;
  3. читать тело запроса чанками (`httpd_req_recv`) → `esp_ota_write`;
  4. `esp_ota_end(handle)` (валидация образа: магия/заголовок/размер);
  5. `esp_ota_set_boot_partition(next)`; ответ `ok`; `esp_restart()` после короткой задержки.
  Ошибки → 400/500 + `esp_ota_abort`. Зарегистрировать `ota_api_start()` в `app_main`.
- Во время заливки контрол-фреймы не идут → watchdog тоже подстрахует «стоп».

## Приложение (iOS)

- **`UpdateClient.swift`** *(new)* (`@MainActor`, async):
  - `latestRelease()` → GET `https://api.github.com/repos/ajohnson-smarthome/AJPicoCar/releases/latest`
    → `(tag, assetURL)` (asset, чьё имя оканчивается на `.bin`).
  - `download(_ url) -> URL` — скачать `.bin` во временный файл (прогресс).
  - `upload(binURL) -> Bool` — `POST CarHost.httpBase + "/ota"` стримом тела файла, прогресс по
    отправленным байтам (через `URLSession upload` + delegate `didSendBodyData`).
- **UI:** в ⚙ Настройки → строка **«Прошивка»** → `FirmwareView`:
  - текущая версия (из `status.fw`), доступная (из GitHub: «актуальна» / «доступно vY»);
  - кнопка **«Обновить»** → шаги: *Скачать (нужен интернет) → Подключись к ESP32-Car → Залить* с
    прогресс-баром → «Перезагрузка…» → апп ждёт, перечитывает `/status`, показывает новую версию.
  - Темы из `Palette`, тексты через `L` (`firmware.*` ключи).

## Mock-сервер (для теста UI в симуляторе)

- `tools/mock_car/mock_car.py`: `POST /ota` — принять тело, залогировать размер, вернуть `ok` (без
  реальной прошивки). Позволяет прогнать поток заливки/прогресса в симуляторе. (GitHub-проверка/скачивание
  тестируются против реального публичного репо.)

## Тестирование

- **Прошивка (host):** чистой логики мало; собрать `idf.py build`; проверить, что разметка валидна и
  образ влезает в `ota_0/ota_1`.
- **На устройстве:** разовый wired-flash dual-OTA → выложить релиз `v1.1` → из аппа обновить → машинка
  перезагрузилась на `v1.1` (`/status` показывает новую версию); проверить откат (битый образ → возврат).
- **Симулятор:** апп видит «доступно vY» (реальный GitHub), скачивает, заливает на мок (`/ota` лог), UI-поток
  и прогресс корректны; обе темы.

## Вне объёма

- Подписанные образы / secure boot (хобби, локальный AP — низкий риск).
- Дельта-обновления, авто-обновление без подтверждения.
- Обновление прошивки C6-компаньона (будущее, при переходе на P4).
