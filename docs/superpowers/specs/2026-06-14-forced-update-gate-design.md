# Обязательный гейт обновления прошивки при запуске — Design

**Дата:** 2026-06-14
**Статус:** дизайн утверждён (обсуждение + визуальный компаньон), готов к плану

## Цель

Приложение жёстко связано с версией прошивки. При запуске нужно гарантировать, что машинка прошита
**актуальной** версией, прежде чем дать управлять. Поскольку проверка/скачивание с GitHub требуют
интернета, а заливка — Wi-Fi машинки (softAP без интернета), вся работа с GitHub делается **заранее**
(пока телефон на интернете), а форс-прошивка — уже на сети машинки из локального кэша.

## Поток запуска (стейт-машина `AppFlow`)

Заменяет текущий корень `ZStack { DriveView; if !online { ConnectView } }`.

```
1. checkInternet      — проба досягаемости GitHub
      ├─ нет ──▶ ЭКРАН «Нет интернета» (блок, «Повторить») ───────────────┐
      └─ есть ──▶                                                          │ retry
2. checkUpdate        — GitHub latest release                              │
      ├─ ошибка API ──▶ ЭКРАН «Не удалось проверить» («Повторить») ────────┘
      └─ ок ──▶ есть кэш .bin нужного билда?
                 ├─ да ──▶ к шагу 4
                 └─ нет ──▶ 3. downloadFirmware — качаем .bin в кэш (прогресс)
4. connectToCar       — ЭКРАН РАДАРА (существующий ConnectView), ждём softAP
      └─ машинка найдена ──▶ bootstrap /status → fw машинки
5. firmwareGate       — сравнить fw машинки vs кэшированный latest (по номеру билда)
      ├─ машинка ≥ latest ──▶ к шагу 6
      └─ машинка < latest ──▶ ЭКРАН «Требуется обновление» (форс-прошивка из кэша,
                                закрыть нельзя) → ребут → перечитать fw → к шагу 6
6. калибровка         — существующий обязательный шит (без изменений)
7. drive              — управление
```

**Порядок гейтов:** Интернет → Обновление скачано → Машинка найдена → Прошивка актуальна → Калибровка → Езда.
Строгий режим (выбор A): нет интернета → всегда блок, даже при наличии кэша (гарантия свежей проверки).

## Экраны

| Экран | Статус | Описание |
|---|---|---|
| **«Нет интернета»** | 🆕 новый | split-layout; слева **Wi-Fi-значок** (концентрические янтарные дуги из общей точки, пульс волной снизу вверх); справа «Нет интернета» · «Подключись к интернету, чтобы проверить обновление прошивки перед поездкой.» · пилюля **«Повторить»** (янтарная). |
| **«Не удалось проверить»** | 🆕 (вариант) | тот же Wi-Fi-значок (или статичный) + «Не удалось проверить обновления» · «Повторить». Отличается от «нет интернета» (GitHub отвечает, но API упал). |
| **Проверка / Скачивание** | ♻️ FirmwareView-визуалы | машинка с волной-пульсом + прогресс; «Проверка обновлений…» → «Скачивание N%». Без «Закрыть». |
| **Подключение к машинке** | ♻️ ConnectView | радар «Поиск машинки…». Без изменений. |
| **«Требуется обновление»** | 🆕 форс-режим FirmwareView | «Прошивка устарела — обнови, чтобы продолжить». Только «Обновить», **без «Позже»/«Закрыть»** (как обязательная калибровка). Заливка из кэша + прогресс + ребут → авто-переход дальше. |
| **Калибровка / Езда** | ♻️ существующие | Без изменений. |

## Кэш и сравнение версий

- **Кэшируем** `.bin` в `Caches/` + в `UserDefaults`: `cachedLatestBuild: Int`, `cachedLatestTag: String`.
- На шаге 2: GitHub latest → если `cachedLatestBuild == latestBuild` и файл на месте → скачивание (шаг 3)
  пропускается; иначе качаем и обновляем кэш.
- **Сравнение** на шаге 5 — существующая `UpdateClient.isUpdateAvailable(running: fwМашинки, latest: cachedLatestTag)`.

## Краевые случаи

- **На GitHub нет версионного релиза** (только легаси `v1.0`, билд-номер = nil): гейт прошивки **не
  активируется** — `latest` без номера → нечего форсить → шаг 5 пропускается, идём к подключению/езде.
  Гейт «оживёт» после первого релиза через `release.sh`. (Шаги 1–3 при этом всё равно проходят: интернет
  проверяется, latest читается; просто «обновление» не форсится.)
- **На машинке прошивка ДО версионирования** (fw без номера) + latest с номером → машинка заведомо старее
  → **форс-обновление** (ловит текущую реальную машинку).
- **GitHub доступен, но API упал/rate-limit** → экран «Не удалось проверить» (отдельно от «нет интернета»).
- **Скачивание/прошивка упали** → ошибка + «Повторить» (как в текущем FirmwareView).
- **Симулятор:** `CarHost` = localhost-мок; интернет-проверка реально стучится на GitHub с Mac; пока latest
  = легаси `v1.0` → гейт пропущен → мок-машинка → езда (не залипает).

## Архитектура и тесты

- **Новый координатор** `AppFlow` (enum фаз + `@MainActor ObservableObject`) заменяет `ZStack` в
  `ESP32CarApp`. Каждая фаза → свой экран; `ConnectView`/`DriveView`/`FirmwareView`-логика переиспользуются.
  `CarStatus`/`CarConnection` запускаются на фазе `connectToCar` (а не сразу при старте).
- **Чистая логика гейта** — host-тест (swiftc + XCTest): `AppFlow.decide(internetOk:latestTag:cachedBuild:carFw:) -> Phase`
  (таблица переходов: нет интернета → noInternet; latest без номера → skip-gate; carFw<latest → updateRequired;
  carFw≥latest или carFw nil-но-latest-nil → drive-path; и т.д.).
- **Новый компонент** `NoInternetView` (split + Wi-Fi-glyph Canvas, пульс через TimelineView).
- **Форс-режим** FirmwareView: параметр `forced: Bool` — скрывает «Закрыть», авто-старт проверки, по
  завершении не закрывается, а сигналит координатору «готово».
- **Симулятор:** проверка последовательности экранов (скриншоты ключевых фаз).

## Вне объёма

- Механика заливки OTA (`/ota`, dual-partition, rollback) и версионирование прошивки — уже сделаны, не меняем.
- Чтение SSID / автодетект сети — нельзя без entitlement; досягаемость определяется пробами (GitHub vs
  192.168.4.1), сети взаимоисключающие.
- Телеметрия по WS / калибровка — без изменений (калибровка остаётся последним гейтом после прошивки).
