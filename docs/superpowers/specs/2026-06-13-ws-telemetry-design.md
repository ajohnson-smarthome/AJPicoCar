# Телеметрия по WebSocket (REST только под конфиг/OTA) — Design

**Дата:** 2026-06-13
**Статус:** дизайн утверждён (обсуждение в чате), готов к плану

## Цель

Устранить архитектурную нестыковку: управление идёт по WS, а телеметрия читается HTTP-поллингом
`/status`. Перенести **живую телеметрию и сигнал живости на WS** (push с борта), оставив REST только
под транзакции (конфиг/OTA) и **один** бутстрап-`/status` при подключении. Побочно: мгновенный детект
обрыва (вместо лага поллинга) и меньше кода в аппе.

## Разделение по фазам/трафику

```
ИНИЦИАЛИЗАЦИЯ (один раз при подключении):
  iOS ──GET /status──▶ машинка → опознание (device/fw) + начальный calibrated → «Поиск» vs «Едем»

РАБОТА (постоянно, один WS-канал /ws):
  iOS ──"t,y" 10 Гц──▶ машинка   (управление — БЕЗ ИЗМЕНЕНИЙ, кормит watchdog)
  iOS ◀── телеметрия 5 Гц ── машинка  (push, JSON)
       живость = свежесть кадров (нет кадра > порога → офлайн)

КОНФИГ/OTA (по требованию): REST /calib* /ramp /trim /ota — БЕЗ ИЗМЕНЕНИЙ
```

**Не трогаем:** управляющий поток (`t,y` 10 Гц), watchdog, REST `/calib*`/`/ramp`/`/trim`/`/ota`.
Это снижает риск для safety-critical пути — только добавляем downstream-пуш и убираем поллинг.

## Формат WS-кадра телеметрии (машинка → апп, 5 Гц)

JSON-текст, только **меняющиеся** живые поля (статичные device/fw остаются в бутстрап-`/status`):
```json
{"rssi":-55,"ws_fps":10,"wdt_trips":0,"uptime_s":123,"heap":198000,"calibrated":true}
```
Направление кадров не конфликтует: апп **шлёт** `"t,y"` (вверх), машинка **шлёт** JSON (вниз);
`ws_handler` на борту по-прежнему принимает только `"t,y"`, апп-receive принимает только телеметрию.

## Прошивка

### `ws_control.{c,h}`
- На WS-handshake (ветка `HTTP_GET` в `ws_handler`) запомнить **сокет клиента**
  `httpd_req_to_sockfd(req)` в `static volatile int s_client_fd` (клиент один — телефон; последний
  коннект перетирает fd).
- **esp_timer 200 мс** (`telemetry_tick`): если `s_client_fd >= 0` → собрать JSON → отправить
  `httpd_ws_send_frame_async(server, s_client_fd, &frame)`. При ошибке отправки → `s_client_fd = -1`
  (пуш молчит до нового коннекта).
- Геттер/паттерн как у существующих счётчиков.

### `status_api.c` (DRY)
- Вынести построение живой части в общий хелпер **`telemetry_json(char *buf, size_t n) -> int`**
  (rssi/ws_fps/wdt_trips/uptime_s/heap/calibrated). Его зовут И `status_get` (добавляя статичные
  device/fw для бутстрапа), И WS-пуш в ws_control. `/status` остаётся полным снимком — апп просто
  перестаёт его поллить.
- `ws_fps`/`rssi` уже считаются в status_api — логика переезжает/делится через хелпер.

## iOS

### `CarConnection.swift`
- receive-loop теперь **парсит** входящий WS-текст как телеметрию: чистая
  `Telemetry.parse(_ json:) -> Telemetry?` (host-тест) → отдать наружу через
  `var onTelemetry: ((Telemetry) -> Void)?` (апп связывает с `CarStatus`).
- Вести `lastTelemetryAt` (для свежести). Управление/реконнект/таймер 10 Гц — без изменений.

### `CarStatus.swift`
- `start()`: **один** `GET /status` (бутстрап — online/calibrated/fw/identity). Сетевой
  таймер-поллинг 1.5 с **убрать**.
- `apply(_ t: Telemetry)`: обновляет rssi/uptimeS/heap/wdtTrips/wsFps/calibrated; ставит
  `online = true`, `lastFrame = now`.
- **Лёгкий freshness-таймер** (локальный, без сети, ~0.5 с): если `now - lastFrame > 1.0 c` →
  `online = false`. Это и есть детект обрыва по свежести.

### `DriveView.swift` / `ESP32CarApp.swift`
- Связать `conn.onTelemetry = { status.apply($0) }`.
- `linkUp` = `status.online` (теперь = бутстрап ок + телеметрия свежа). Палочки по RSSI — как сейчас.
- **Пилюля по свежести:** на 5 Гц «живое» число мс было бы шумом (0–200 мс), поэтому показываем
  свежесть **в полезную сторону** — пока свежо: «На связи»; когда устарело (> порога): фейд + «Поиск…»
  (то же место, где раньше было «N мс»). Внутренняя метрика — возраст последнего кадра — рулит online/offline.
- `ConnectView` гейтится на бутстрап-`/status` (как сейчас); дальше живость — WS.

## Мок

`tools/mock_car/mock_car.py`: добавить **5 Гц пуш телеметрии** в `/ws` (сейчас мок только принимает) —
JSON с теми же полями (rssi/ws_fps/wdt_trips/uptime_s/heap/calibrated). `/status` остаётся как бутстрап.

## Тестирование

- **C:** host-тест `telemetry_json()` — формат/границы буфера.
- **iOS:** нативный swiftc-тест `Telemetry.parse` (валидный/битый JSON, частичные поля) + зеркало в XCTest.
- **Симулятор:** мок пушит 5 Гц → апп показывает живые rssi/uptime/calibrated **без** `/status`-поллинга;
  обрыв WS → офлайн по свежести за ~1 с; реконнект → снова живые. `/status` дёргается ровно один раз.
- Сборка прошивки + iOS; grep кириллицы/цветов.

## Вне объёма

- Управляющий протокол `"t,y"`, watchdog, REST `/calib*`/`/ramp`/`/trim`/`/ota` — без изменений.
- Настоящий round-trip-пинг (отвергнут — свежести достаточно).
- Бинарный формат кадра (JSON проще и консистентнее; 5 Гц × ~120 Б — мелочь).
