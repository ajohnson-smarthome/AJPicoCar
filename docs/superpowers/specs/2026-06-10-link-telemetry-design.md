# Телеметрия связи (RSSI + здоровье управления) — Design

**Дата:** 2026-06-10
**Статус:** дизайн утверждён (обсуждение в чате), готов к плану

## Цель

Машинка отдаёт в `/status` телеметрию радиоканала и здоровья управления, которой нет у телефона:
**реальный RSSI** клиента (то, что iOS не даёт без entitlement — меряем с другого конца линка),
**частоту WS-кадров** и **счётчик срабатываний watchdog**. Апп переводит палочки сигнала на
настоящий RSSI (пинг — fallback) и показывает предупреждение при wdt-срабатываниях.

## Прошивка

### Расширение `/status` (`main/status_api.c`)
Новые поля (JSON остаётся одной строкой, буфер увеличить до ~224):
```json
{"device":"esp32-car","fw":"v1.1","uptime_s":123,"calibrated":true,"heap":200000,
 "rssi":-55,"ws_fps":10,"wdt_trips":0}
```
- **`rssi`** — `esp_wifi_ap_get_sta_list()` → RSSI первой подключённой станции (наш единственный
  клиент); если станций нет (нештатно для запроса по WiFi) → `0` (апп трактует 0 как «нет данных»).
  REQUIRES уже содержит `esp_wifi`.
- **`ws_fps`** — целое: кадры/сек между двумя последовательными `GET /status`
  (в `status_api.c` статически хранится прошлое значение счётчика + штамп `esp_timer`; при первом
  запросе/паузе >10 с → 0).
- **`wdt_trips`** — счётчик аварийных стопов с загрузки.

### Счётчики
- `main/ws_control.{c,h}`: атомарный `static volatile uint32_t s_frames`; `++` на каждом валидном
  кадре (рядом с `watchdog_feed()`); геттер `uint32_t ws_control_frames(void)`.
- `main/watchdog.{c,h}`: `static volatile uint32_t s_trips`; `++` в `wdt_cb` при срабатывании;
  геттер `uint32_t watchdog_trips(void)`.
(32-бит volatile на одноядерном RISC-V читаются атомарно — как существующий `s_last_feed_ms`.)

## iOS

- **`CarStatus`**: парсить `rssi` (Int?, 0 → nil), `wdtTrips` (Int?), `wsFps` (Int?, пока без UI).
- **`ControlModel.signalLevel`** — новая чистая перегрузка (хост-тест):
  `signalLevel(online:rssi:pingMs:)`: офлайн → 0; **RSSI есть** → пороги
  `≥ -50` → 4, `≥ -60` → 3, `≥ -70` → 2, иначе 1; **RSSI нет** → старая пинг-логика (50/120/250 мс).
- **`DriveView`**: `signalLevel` использует `status.rssi`; в нижнюю статус-строку добавляется
  **четвёртый элемент только при `wdtTrips > 0`**: иконка `exclamationmark.triangle` (warn) + число —
  «связь рвалась N раз». Строка `drive.wdtTrips` = "обрывов: %d".
- **Мок**: `/status` дополняется `"rssi": -58, "ws_fps": 10, "wdt_trips": 0`.

## Тестирование

- Хост (swiftc): `signalLevel` — RSSI-пороги, fallback на пинг, офлайн.
- Прошивка `idf.py build`; host-тесты C не затронуты (счётчики не pure).
- Симулятор: палочки из мокового RSSI −58 (→ 3 палки); поставить мок `wdt_trips: 3` → в статус-строке
  появляется «⚠ обрывов: 3»; вернуть 0 → исчезает.
- Железо (позже): отходить с телефоном от машинки → палочки падают по-настоящему.

## Вне объёма

- История/графики RSSI, latency кадров (нет таймстампов в протоколе), UI для `ws_fps` (поле есть — на будущее).
