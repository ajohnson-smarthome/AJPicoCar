# 4WD RC-машинка с танковым разворотом (WiFi + веб-пульт)

**Дата:** 2026-06-07
**Статус:** дизайн утверждён, готов к плану реализации

## Цель

Превратить текущую прошивку (бинарное управление 4 моторами через USB-консоль)
в полноприводную RC-машинку с:
- управлением по WiFi с iPhone (веб-UI/PWA, без App Store, Mac и Xcode);
- плавной пропорциональной скоростью (ШИМ 0..100%);
- танковым разворотом (skid steer) — разворот на месте и поворот дугой;
- двумя схемами управления (аркадная — 1 стик, танковая — 2 стика);
- обязательной калибровкой моторов при первом подключении.

## Ключевые решения (из брейншторма)

| Вопрос | Решение |
|---|---|
| Транспорт | WiFi softAP на машинке + WebSocket + captive-portal |
| Сторона телефона | Веб-UI/PWA, отдаётся прошивкой; «добавить на экран» |
| Скорость | Плавная, пропорциональная (duty 0..4095) |
| Схемы управления | Обе: аркадная и танковая (переключатель в UI) |
| Калибровка | Обязательный первый экран, если не выполнена; 4 угла + направление |
| Режим кручения при калибровке | Короткие импульсы (~0.5с) на низкой скорости, кнопка «ещё раз» |
| Безопасность | Watchdog-автостоп при потере связи |

## Архитектура

```
┌─────────────┐   WiFi softAP    ┌──────────────────────────────────┐
│  iPhone      │   "ESP32-Car"    │  ESP32-C6                         │
│  Safari/PWA  │◄────────────────►│  ┌────────────┐                  │
│  пульт / cal │   WebSocket {t,y}│  │ HTTP+WS    │                  │
│              │   + captive DNS  │  │ server     │──┐               │
└─────────────┘                   │  └────────────┘  ▼               │
                                   │  ┌────────────┐  ┌────────────┐  │
                                   │  │ watchdog   │  │ mixer      │  │
                                   │  │ (автостоп) │  │ t,y→L,R    │  │
                                   │  └─────┬──────┘  └─────┬──────┘  │
                                   │        ▼               ▼         │
                                   │  ┌──────────────────────────┐    │
                                   │  │ motors: ramp + calib-map  │    │
                                   │  │        + shoot-guard      │    │
                                   │  └─────────────┬─────────────┘    │
                                   │                ▼ I2C               │
                                   │   PCA9685 → 4× BTS7960 → 4 мотора  │
                                   │   NVS: таблица калибровки          │
                                   └──────────────────────────────────┘
```

**Принцип:** телефон шлёт только нормализованное «намерение» (`throttle`, `yaw`).
Вся логика разворота, скорости, плавности, безопасности и маппинга — на ESP.
Телефон остаётся «тупым», протокол единый для обеих схем и будущего нативного аппа.

## Протокол управления

- **Канал:** WebSocket поверх softAP.
- **Формат:** компактный текст `t,y`, где `t,y ∈ [-1.0, 1.0]`, частота ~30 Гц.
- **Калибровочные команды:** отдельные WS-сообщения (или REST), например
  `cal:spin:<channel_pair>` — крутануть указанный мотор вперёд импульсом.
- Телеметрия назад по WS: heartbeat «связь жива» (для индикатора в UI).

## Математика танкового разворота (чистая функция, тестируется на хосте)

```
left  = t + y
right = t - y
m = max(|left|, |right|, 1.0)   # нормализация с сохранением пропорции
left /= m ;  right /= m         # оба в [-1, 1]
```

Проверочные случаи:
- `t=+1, y=0`   → left=+1, right=+1  → прямо
- `t=0,  y=+1`  → left=+1, right=-1  → разворот на месте (танк)
- `t=+0.5,y=+0.5`→ left=1, right=0   → поворот дугой
- `t=-1, y=0`   → оба -1             → назад

Сведение схем телефона к `(t,y)`:
- **Аркада (1 стик):** `t = stickY`, `y = stickX`.
- **Танк (2 стика):** `t = (L+R)/2`, `y = (L−R)/2` (обратное смешивание, round-trip честный).

## Скорость, плавность, безопасность

**Борт → мотор** (скорость борта `s ∈ [-1,1]` применяется к обоим моторам борта):
- `s > +deadzone` → вперёд: CH_A = `|s|·4095`, CH_B = 0
- `s < −deadzone` → назад:  CH_A = 0, CH_B = `|s|·4095`
- иначе → стоп (оба 0)
- Защита от shoot-through сохраняется (никогда оба канала ≠ 0).

**Ramp (slew-rate limit):** duty меняется не более чем на ΔX за управляющий тик —
гасит рывки и снижает риск брауна-аута (пусковой ток 5–10× номинала).

**Watchdog:** таймер ~50 Гц. Если `now − last_ws_msg > 300 мс` или WS закрыт →
плавный сброс всех моторов в 0. Без живого пульта машина не едет.

*Конкретные значения `deadzone`, `ΔX`, порога watchdog — стартовые, подбираются на стенде.*

## Калибровка моторов

### Что определяем для каждого из 4 моторов (пары CH0/1, CH2/3, CH4/5, CH6/7)
1. **Позицию:** FL (перёд-лево), FR (перёд-право), RL (зад-лево), RR (зад-право).
2. **Направление:** совпадает ли «forward»-команда с реальным вращением колеса
   вперёд; иначе знак инвертируется.

Результат — таблица `позиция → {channel_pair, sign}`. Миксер берёт из неё
`LEFT = {FL, RL}`, `RIGHT = {FR, RR}` и правильные знаки. Хардкод-маппинга бортов нет.

### UX-поток (drive-then-tag)
До калибровки ESP не знает расположения колёс, поэтому «крутим → пользователь отмечает»:

```
Шаг N/4:  [ESP крутит мотор на CHx/y вперёд короткими импульсами ~0.5с]
          "Какое колесо крутится и куда?"
          Позиция:  [ FL ] [ FR ] [ RL ] [ RR ]
          Вращение: ( ) вперёд   ( ) назад
          [Крутануть ещё раз]   [Дальше →]
... повторяется для всех 4 пар каналов
Готово:   [✓ все 4 угла назначены уникально] → Сохранить
```
UI запрещает назначить один угол дважды и требует все 4 угла перед сохранением.
Кручение — короткими импульсами на низкой скорости (безопасно на стенде),
под тем же watchdog/shoot-guard.

### Персистентность и гейт первого экрана
- Таблица калибровки + флаг `calibrated=true` хранятся в **NVS** (переживают перезагрузку).
- При загрузке страницы JS запрашивает `GET /calib`. Если не откалибровано →
  **первый и единственный доступный экран — калибровка**; пульт заблокирован.
- После сохранения → автоматически открывается пульт.
- В пульте — кнопка «⚙ Перекалибровать» (на случай переподключения моторов).

## Структура прошивки (рефактор `main/main.c`)

| Модуль | Ответственность | Тестируемость |
|---|---|---|
| `pca9685.c/.h` | I2C-драйвер PWM (вынести из текущего main) | железо |
| `mixer.c/.h` | чистые ф-ии t,y→L,R, clamp, deadzone | **хост, юнит-тесты** |
| `motors.c/.h` | борт→мотор, ramp, применение calib-таблицы, shoot-guard | хост (мок PWM) |
| `calibration.c/.h` | хранение/чтение NVS, валидация уникальности 4 углов | хост (мок NVS) |
| `control_server.c/.h` | softAP, HTTP, WebSocket, captive DNS, эндпоинты /calib | железо |
| `main.c` | сборка: init → server → loop | — |
| `web/` | `index.html`, `app.js`, `style.css` — вшиты через `EMBED_FILES` | браузер |

Старая USB-консоль остаётся как **отладочный fallback** (калибровка/диагностика,
совместимость с мостом `/tmp/esp_in`).

## Веб-UI (PWA + captive-portal)

- `index.html` + JS вшиты в прошивку, отдаются с `192.168.4.1`.
- **Captive-portal:** DNS-сервер на ESP резолвит всё в `192.168.4.1` + ответ на
  iOS-проверку (`/hotspot-detect.html`) → Safari сам открывает пульт при подключении
  к WiFi `ESP32-Car`.
- **PWA:** манифест + `apple-mobile-web-app-capable` → «Добавить на экран», запуск
  без адресной строки.
- Экраны: **калибровка** (гейт), **пульт** с переключателем аркада/танк, тач-джойстики
  на canvas, индикатор связи (heartbeat по WS), кнопка «Перекалибровать».
- Landscape, крупные зоны под большой палец.

## Тестирование

- **Юнит-тесты на хосте (TDD):** `mixer` (clamp, разворот на месте, deadzone,
  нормализация), `motors` (применение sign-таблицы, ramp, shoot-guard),
  `calibration` (отказ при дубле угла, при <4 моторах).
- **Стендовая интеграция:** колёса вывесить, гонять с телефона, смотреть телеметрию;
  мост `/tmp/esp_in` для диагностики.
- **Полевой тест:** iPhone → `ESP32-Car` → калибровка → езда; проверка watchdog
  (унести телефон / заглушить WiFi → автостоп).

## Фазы реализации (инкременты)

1. **Рефактор + ядро без WiFi:** вынести `pca9685`; добавить `mixer` + `motors` с
   пропорциональной скоростью; хост-тесты; проверка через USB-мост.
2. **WiFi softAP + HTTP-сервер:** поднять сеть `ESP32-Car`, отдать статическую страницу.
3. **WebSocket + протокол `t,y` → моторы:** живое управление с телефона.
4. **Watchdog + ramp:** безопасность и плавность.
5. **Калибровка:** `calibration` модуль + NVS + эндпоинты + калибровочный экран как гейт.
6. **Captive-portal + PWA + обе схемы UI:** полировка, «как приложение».

## Открытые параметры (подбор на стенде, не блокируют старт)

- `deadzone`, шаг ramp `ΔX`, порог watchdog (старт 300 мс).
- Скорость/длительность импульса калибровки.
- Имя/пароль softAP `ESP32-Car`.

## Вне объёма (будущие идеи)

- Нативное iOS-приложение с авто-подключением к WiFi (NEHotspotConfiguration).
- Второй ESP32 как физический пульт по ESP-NOW.
- Телеметрия напряжения батареи, индикатор заряда.
- Поколёсное/поосевое распределение тяги и трим (torque bias, anti-scrub в повороте,
  компенсация слабого мотора) — заложено 4-угловой калибровкой. Настоящее руление
  осями (Ackermann/4WS) или краб/строуф потребуют доп. железа (серво или меканум-колёса) —
  на текущей сборке (4 жёстких колеса, skid steer) физически недоступны.
