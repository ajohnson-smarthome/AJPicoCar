# Дальнобойная связь (0.5–1 км, видео+лидар, низкая задержка, iPhone) — ресёрч

**Дата:** 2026-06-10 · Deep-research: 23 источника, 90 claims, 25 верифицировано (**19 ✓ / 6 ✗**).
Вводные пользователя: дальность **0.5–1 км**, **видео на всю дальность**, приоритет — **минимальная
задержка**, iPhone — конечный пульт (допускается наземный мост).

## Главный вывод

Под «1 км + HD-видео + низкая задержка» оптимум — **цифровой WiFi-broadcast видеолинк**
(WFB-NG / OpenIPC-класс) + **отдельный дальнобойный канал управления (ELRS)**, а картинка попадает
в iPhone **через наземную станцию** (мини-Linux, декодирует и ре-стримит в локальный WiFi нашему
приложению). Сотовый путь даёт неограниченную дальность, но задержка в разы выше цели.

## ❗ Жёсткое ограничение: iOS не принимает такой радиоканал напрямую (3-0)

Все декодеры этих стеков — **только Android** и требуют USB-приёмник RTL8812au, который iOS хостить
не умеет: **PixelPilot** (Android-only, в т.ч. Meta Quest), **RubyFPV** на телефон — только Android
через USB-tethering. Значит наземный мост (Raspberry Pi + USB-WiFi-карта + антенна) — **обязателен,
а не опционален**: он декодирует широковещание и ре-стримит в наш iOS-апп по локальному WiFi.

## Подтверждённые факты

1. **WFB-NG** — ядро опенсорсного дальнолинка: дешёвые карты RTL8812au/eu в **monitor mode** шлют
   сырые 802.11-пакеты **без association и ACK** → обходят лимиты обычного WiFi по дальности/задержке.
2. **WFB-NG несёт H.264 RTP как UDP, 1:1 RTP→802.11** (минимум задержки) + **IPv4-туннель для
   телеметрии лидара** на том же линке. Дефолт ~7 Мбит/с = 720p@49fps (4 Мбит/с) + 2 телеметрии.
   Видео — сырой UDP, НЕ через туннель.
3. **OpenIPC/RubyFPV: ~40 мс glass-to-glass** (независимый тест 1000fps-камерой — 25 мс, как DJI
   FPV V1). Настоящий низколатентный HD на дистанции ~1 км. (2-1)
4. **Наземная станция = мини-Linux SBC** (Pi Zero 2 W / Pi 2-4 / Radxa). Декодирует и может
   ре-экспортировать H.264 по UDP→RTP/RTSP через GStreamer — это и есть механизм отдать видео
   в локальный WiFi нашему iOS-аппу.
5. **ELRS — рекомендованный отдельный канал УПРАВЛЕНИЯ** (~2-4 мс OTA). Но обратная телеметрия
   скромная: ~5000 bps лучший случай (500Гц), падает до 78-250 bps на дальних режимах → руль +
   мелочь ок, **непрерывный лидар — нет** (лидар лучше через IPv4-туннель WFB-NG).
6. **ELRS AirPort** — прозрачный двунаправленный serial-мост (несёт любые данные), потолок ~5000 Б/с
   OTA (короткая дистанция); на 0.5-1 км реально ~1665 Б/с, без ACK/ретрансмиссии. RC + AirPort
   одновременно = нужна 2-я пара TX/RX.
7. **Сотовый (4G/LTE + релей)**: дальность по вышкам, но FPV-задержка заметно хуже — управление
   ~97 мс (peer-reviewed, < 150 мс), видео ≥ 300 мс (MJPEG) / H.264 +0.7-0.8 с (артефакт буфера
   декодера, лечится до < 100 мс). На порядок выше цели 30-50 мс. ELRS для сравнения ~2-4 мс.
8. **ESP32-нативный видеолинк существует — `hx-esp32-cam-fpv`**: сам ESP32 шлёт JPEG-кадры через
   packet injection + FEC (k=6/n=12), в стиле wifibroadcast. Дешёвое продолжение текущего ESP32-стека
   (НЕ полноценный WFB-NG, MJPEG, тоже нужен Pi/Radxa-приёмник).

## Опровергнуто (маркетинг / непроверено)

- ✗ WFB-NG «~20 км с патч-антенной» (1-2) — реальная дальность зависит от антенн/мощности/LoS,
  цифры на 1 км в поле никто не подтвердил.
- ✗ Sky Drone «видео < 150 мс по 4G», «неограниченная дальность», «iOS-клиент есть» — **все три
  опровергнуты** (1-2 / 0-3 / 0-3). Считать маркетингом.

## Три реалистичные архитектуры (синтез)

| | Что | Задержка | Дальность | Сложность | Ориентир цены |
|---|---|---|---|---|---|
| **A. WFB-NG + ELRS + Pi-мост** ⭐ | OpenIPC-видео + ELRS-управление, Pi ре-стримит в iOS-апп | видео ~40 мс (+форвард-хоп Pi), управление ~2-4 мс | 0.5-1 км+ (антенны) | высокая | Pi 4/5 ~$60-80 + RTL8812au ~$25-40 + антенна + ELRS TX ~$30-50 / RX ~$15-25 |
| **B. ESP32-P4 broadcast + Pi-мост** | P4 шлёт H.264/MJPEG (hx-esp32-cam-fpv-стиль), Pi ре-стримит | низкая, чуть хуже A | короче A | средняя (всё на ESP) | дешевле A, без отдельного air-unit |
| **C. 4G на борту + WebRTC в iOS** | SIM7600/EC25 + релей-сервер + WebRTC | ~100-300 мс (не «твич») | не ограничена вышками | средняя | модем ~$50-80 + сервер |

Все три **требуют наземный мост** (Pi + RTL8812au + направленная антенна), т.к. iOS не ест RF напрямую.

## Как стыкуется с нашим стеком

- iOS-апп **не меняем радикально**: мост отдаёт видео по локальному WiFi (RTP/RTSP/WebRTC) + наши
  WS-управление/REST остаются. Возможно, в апп добавится видео-вью (декод H.264 — у iOS аппаратный).
- ESP32-P4 на борту остаётся камерой+лидаром+мотор-контроллером; меняется только **радиотракт**
  до моста (вместо softAP — broadcast/air-unit), плюс ELRS-приёмник на UART для управления.
- Лидар → IPv4-туннель WFB-NG (не ELRS).

## Открытые вопросы

1. Реальная дальность WFB-NG/OpenIPC у земли (высота машинки, multipath) на 5.8 vs 2.4 ГГц для 1 км.
2. Сколько мс добавляет форвард-хоп Pi-моста (ре-стрим в локальный WiFi) и съест ли наш WS/REST-апп
   RTP/H.264 без редизайна.
3. Можно ли аппаратный H.264 P4-Nano скормить packet-injection TX напрямую (вместо MJPEG hx-esp32).
4. Есть ли поддерживаемый мост с WebRTC/низколатентным RTSP-эндпоинтом, который iOS ест нативно
   (обойти Android-lock декодеров).
